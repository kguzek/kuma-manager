# Kuma Manager Agent Guidelines

## Architecture

- Keep exported TypeScript types in `src/types/`.
- Use `import type` for type-only imports.
- Keep Kuma API access behind `src/api/kuma/client.ts`; UI code must not call Socket.io directly.
- Keep browser storage helpers in `src/utils/`.
- Prefer feature folders for product code: `src/features/<feature>/components`, `src/features/<feature>/hooks`, and `src/features/<feature>/utils`.
- Keep shared reusable components in `src/components/`.
- Keep each product component in its own file. Shadcn registry primitives may export related primitives together when generated by the registry, but new app components should stay one component per file.
- Keep `App.tsx` as orchestration only: session state, top-level routing, and cross-feature coordination.

## React

- Use TypeScript with strict types for props, state, and API responses.
- Prefer simple local state before adding context or external state libraries.
- Use React Hook Form for forms.
- Use Shadcn `Field` primitives for accessible form layout.
- Do not use the old Shadcn `Form` component pattern.
- Avoid `useMemo` and `useCallback` unless they solve a concrete performance or referential-stability issue.
- Keep effects focused on synchronization with external systems: browser history, storage, network sessions, or subscriptions.

## Accessibility

- Use real HTML links (`<a href="...">`) for navigation, not click-only buttons or divs.
- When navigation should look like a Shadcn button, render `Button` with `asChild` and put the link inside it.
- Preserve standard browser behaviors for links, including open-in-new-tab and modifier-click behavior.
- Keep global status/alert messaging in one page-level location; avoid duplicating progress alerts inside cards.

## Styling

- Use Tailwind CSS v4 utilities and theme tokens.
- Preserve the dark-first design language.
- Use Shadcn components for common UI primitives.
- Use Lucide icons consistently for actions and page affordances.
- Prefer minimal layout shifts for inline edit states.

## Kuma Sync Rules

- Only tags starting with `monitor:` opt a monitor into sync.
- Never sync heartbeat/history data.
- Treat monitor IDs as instance-local implementation details.
- Route synced monitor detail pages by tag suffix, e.g. `/monitors/example.com` maps to `monitor:example.com`.
- All Kuma API calls should go through the adapter so internal API breakage is isolated.

## Quality

- Run `pnpm build` after structural or TypeScript changes.
- Keep changes small and composable.
- Do not introduce backwards-compatibility shims unless there is a concrete need.
- Prefer clear file names over generic names like `helpers.ts`.
